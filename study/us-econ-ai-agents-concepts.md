# US ECON AI Agents Concept Notes

## 한 줄 컨셉

US ECON AI는 "AI가 읽어주는 미국 경제 대시보드"입니다. 숫자는 Trading Economics처럼 빠르게 보여주고, 해석은 KOTRA 리포트처럼 한국 기업과 투자자 관점에서 풀어줍니다.

첫 화면의 목표는 단순합니다.

```text
사용자가 10초 안에 오늘 미국 경제가 좋은지, 나쁜지, 애매한지 이해한다.
```

## OpenAI Agents 문서에서 가져올 핵심

OpenAI Agents SDK 문서는 에이전트를 "계획하고, 도구를 호출하고, 여러 전문가와 협업하고, 여러 단계 작업을 끝낼 만큼의 상태를 유지하는 애플리케이션"으로 설명합니다. 이 프로젝트에서는 그 개념을 하나의 거대한 AI에게 모두 맡기는 방식이 아니라, 역할이 분명한 여러 경제 분석 에이전트가 협업하는 구조로 해석합니다.

에이전트 하나는 보통 다음을 묶은 단위입니다.

- `name`: 추적 화면과 로그에서 보이는 역할 이름
- `instructions`: 이 에이전트가 해야 할 일, 말투, 금지사항
- `model`: 사용할 모델과 추론 성향
- `tools`: 경제 지표 API, 뉴스 검색, 리포트 검색, 차트 데이터 조회 같은 실행 능력
- `handoffs`: 다른 전문 에이전트에게 일을 넘기는 규칙
- `outputType`: 화면과 백엔드가 안정적으로 받을 수 있는 구조화된 결과
- `guardrails`: 잘못된 숫자, 출처 누락, 투자 조언 과잉을 막는 검증 규칙

초보자에게 쉽게 말하면, 에이전트는 "프롬프트가 붙은 함수"보다 큽니다. 함수가 한 번 계산하고 끝난다면, 에이전트는 필요한 자료를 찾고, 다른 담당자에게 물어보고, 검증을 통과한 뒤, 사용자가 이해할 수 있는 결과를 만듭니다.

전문가 관점에서는 에이전트를 "모델 호출 + 도구 실행 + 상태 관리 + 관찰 가능성 + 정책 제어를 묶은 워크플로 단위"로 보는 것이 정확합니다.

## 이 제품의 에이전트 구조

처음부터 복잡한 멀티 에이전트로 시작할 필요는 없습니다. MVP에서는 "경제 브리핑 매니저" 하나가 전체 답변을 책임지고, 나머지는 도구 또는 하위 전문 에이전트처럼 작동하는 구성이 가장 안정적입니다.

```text
경제 브리핑 매니저
-> 거시지표 수집
-> 시장지표 수집
-> 경제일정 정리
-> 뉴스/리포트 요약
-> 한국 영향 해석
-> 출처/숫자 검증
-> 최종 AI 요약 생성
```

### 경제 브리핑 매니저

사용자에게 보이는 최종 답변을 책임집니다. "오늘 미국 경제는 어떤 상태인가?"라는 질문에 한 문단으로 답하고, 근거 지표 링크를 반드시 붙입니다.

좋은 출력 예시는 다음 흐름입니다.

```text
미국 경제는 고용은 아직 버티지만 소비와 물가 압력이 엇갈리는 상태입니다.
근거: CPI, 실업률, 비농업고용, 소매판매, 10년물 금리, 달러인덱스
한국 영향: 환율 부담 상승, 수출 업종별 수요 차별화, 금리 민감 업종 주의
```

### 거시지표 수집 에이전트

GDP 성장률, 기준금리, CPI, Core CPI, PCE, 실업률, 비농업고용, 소매판매, 산업생산, 무역수지, 경상수지, 정부부채/GDP를 담당합니다.

이 에이전트의 핵심은 "숫자 자체"보다 "숫자의 신뢰 조건"입니다. 모든 값은 기준일, 단위, 출처, 이전치 대비 변화가 함께 있어야 합니다. 예를 들어 CPI는 전월 대비인지 전년 대비인지, 계절조정 값인지, 발표치인지 수정치인지가 중요합니다.

### 시장지표 수집 에이전트

S&P 500, Nasdaq, Dow, 미국 10년물/2년물 금리, 달러인덱스, USD/KRW, WTI, Brent, 금, 주요 기술주를 담당합니다.

시장지표는 실시간성이 강하기 때문에 "현재가"만 보여주면 위험합니다. 기준 시각, 전일 대비 변화, 변동률, 통화 단위, 데이터 지연 여부를 같이 보여줘야 합니다.

### 경제 캘린더 에이전트

FOMC, CPI, 고용보고서, GDP, PCE, 소매판매, ISM PMI 일정을 담당합니다.

각 이벤트는 다음 구조로 저장하는 것이 좋습니다.

```text
발표일
중요도
이전치
예상치
실제치
해석
출처
관련 지표 링크
```

실제치가 아직 없으면 "예정" 상태로 표시합니다. 실제치가 나온 뒤에는 예상 대비 상회/하회, 이전치 대비 개선/악화, 시장 반응 가능성을 분리해서 적습니다.

### 뉴스/리포트 에이전트

미국 경제 뉴스, Fed 발언, 공급망 이슈, 관세/통상 규제, 산업별 리포트를 요약합니다.

뉴스 요약은 의견이 섞이기 쉽기 때문에 "무슨 일이 있었나", "왜 중요한가", "한국 기업에 주는 시사점"을 분리합니다. 특히 관세, 수출 규제, 보조금, 반도체/자동차/배터리 정책은 한국 사용자에게 별도 표시해야 합니다.

### 한국 영향 해석 에이전트

한국 사용자에게 중요한 축은 다음 다섯 가지입니다.

- 수출: 미국 소비와 산업생산이 한국 수출 수요에 주는 영향
- 환율: 달러인덱스, USD/KRW, 금리차가 원화와 수입 비용에 주는 영향
- 금리: 미국채 금리와 Fed 방향이 국내 금리, 성장주, 부동산 심리에 주는 영향
- 소비: 미국 소매판매와 고용이 한국 소비재, 자동차, 전자 수요에 주는 영향
- 관세/규제: 통상 규제와 산업 정책이 한국 기업 공급망에 주는 영향

이 에이전트는 경제학 교과서식 설명보다 "그래서 한국 기업이나 투자자는 무엇을 봐야 하나"에 집중해야 합니다.

### 출처/숫자 검증 에이전트

AI 요약이 멋있어도 숫자가 틀리면 제품 신뢰가 무너집니다. 이 에이전트는 화면에 나가는 모든 숫자가 다음 조건을 만족하는지 확인합니다.

- 기준일이 있는가
- 단위가 있는가
- 출처가 있는가
- 이전치 대비 변화가 있는가
- AI 요약의 근거 링크가 있는가
- 실제치와 예상치를 혼동하지 않았는가
- 발표 전 일정과 발표 후 결과를 구분했는가

OpenAI Agents 문서의 guardrails와 human review 개념은 여기에 잘 맞습니다. 자동 검증으로 막을 수 있는 것은 막고, 투자 판단이나 민감한 통상 이슈처럼 위험한 해석은 사람이 검토할 수 있게 설계합니다.

## 첫 화면 정보 구조

카드를 많이 늘어놓는 방식보다, 넓은 정보 구역을 몇 개만 두는 것이 좋습니다.

### 1. AI 요약 영역

화면 최상단에는 한 문단 요약을 둡니다.

```text
오늘 미국 경제 상태: 견조하지만 물가와 금리 부담이 남아 있음
근거 지표: CPI, 실업률, 비농업고용, 소매판매, 10년물 금리
한국 영향: 환율 부담 중립~상승, 수출 업종별 차별화, 금리 민감 업종 주의
```

요약은 반드시 근거 지표 링크와 연결되어야 합니다. 사용자가 문장을 클릭하면 어떤 지표 때문에 그런 판단을 했는지 볼 수 있어야 합니다.

### 2. 핵심 지표 스트립

5~8개 핵심 지표를 한 줄 또는 두 줄로 보여줍니다.

추천 지표는 다음입니다.

- GDP 성장률
- 기준금리
- CPI 또는 Core CPI
- PCE
- 실업률
- 비농업고용
- 소매판매
- 미국 10년물 금리 또는 USD/KRW

각 지표는 값, 단위, 기준일, 출처, 이전치 대비 변화를 함께 보여줍니다.

### 3. 경제 캘린더 로그

경제 일정은 일반 달력보다 "실시간 로그 패널"처럼 보이게 하면 제품 성격과 잘 맞습니다.

```text
[중요] 2026-06-15 소매판매
이전치: 0.1% | 예상치: 0.3% | 실제치: 발표 전
해석: 소비 탄력이 확인되면 금리 인하 기대가 약해질 수 있음
```

### 4. 시장지표 미니 대시보드

S&P 500, Nasdaq, Dow, 미국채 10년/2년, 달러인덱스, USD/KRW, WTI/Brent, 금을 압축해서 보여줍니다.

전문가용 화면처럼 너무 많은 숫자를 넣기보다, "위험 선호", "금리 압력", "달러 강세", "원자재 부담" 네 가지 상태로 묶어 보여주면 이해가 빠릅니다.

### 5. 미국 경제/통상 리포트

뉴스 카드가 아니라 리포트 피드로 보이게 만듭니다.

각 항목은 다음 구조가 좋습니다.

```text
제목
분류: Fed / 물가 / 고용 / 통상 / 공급망 / 산업
핵심 요약
한국 기업 시사점
관련 지표
출처
```

### 6. 관심 지표/알림

사용자는 모든 지표를 매일 보지 않습니다. USD/KRW, 10년물 금리, CPI, 소매판매, 반도체 관련 통상 뉴스처럼 관심 지표를 저장하고 알림을 받을 수 있어야 합니다.

## 지표 상세 페이지

상세 페이지는 단순 차트 페이지가 아니라 "한 지표를 이해하는 작은 리포트"여야 합니다.

필수 구성은 다음입니다.

- 1년, 5년, 10년 차트
- 최근 발표 내역
- 이전치, 예상치, 실제치
- AI 해석
- 관련 리포트
- 출처와 기준일
- 한국 영향

예를 들어 CPI 상세 페이지는 "물가가 높다"에서 끝나면 부족합니다. Core CPI와 PCE의 차이, Fed가 왜 PCE를 중요하게 보는지, 물가가 USD/KRW와 미국채 금리에 어떤 식으로 연결되는지까지 설명해야 전문가도 납득합니다.

## 데이터 모델 초안

숫자 신뢰성을 위해 지표 데이터는 처음부터 구조화해야 합니다.

```text
MetricSnapshot
- id
- name
- category
- value
- unit
- period
- baseDate
- sourceName
- sourceUrl
- previousValue
- change
- changePercent
- interpretation
- updatedAt
```

경제 일정은 별도 모델이 좋습니다.

```text
EconomicEvent
- id
- title
- releaseDateTime
- importance
- previousValue
- forecastValue
- actualValue
- unit
- status
- interpretation
- sourceName
- sourceUrl
- relatedMetricIds
```

AI 요약은 근거 링크를 잃지 않도록 별도 구조를 둡니다.

```text
AiBrief
- summary
- statusLabel
- evidenceMetricIds
- evidenceEventIds
- koreaImpact
- risks
- generatedAt
```

## 디자인 방향

이 제품은 금융 터미널처럼 차갑기만 하면 어렵고, 일반 뉴스 사이트처럼 가볍기만 하면 신뢰가 약합니다. 어두운 배경 위에 에메랄드, 민트, 유리질 광택을 쓰되, 정보 밀도는 정제되어야 합니다.

추천 원칙은 다음입니다.

- 배경은 어둡게, 주요 숫자는 밝게, 상승/하락 색은 일관되게
- 카드 남발 대신 큰 정보 영역 4~6개로 구성
- AI 요약은 화면의 중심이지만 근거 지표보다 과장되어 보이면 안 됨
- 타임라인은 경제 일정과 뉴스가 섞이는 실시간 로그 패널처럼 표현
- 스크롤 시 은은한 reveal과 glow를 사용하되, 숫자 가독성을 방해하지 않음
- 모바일에서는 사이드바를 하단 또는 접힘 메뉴로 바꾸고, 지표 스트립은 가로 스크롤보다 2열 그리드가 안전

전문가가 보는 화면에서는 화려함보다 일관된 기준이 더 중요합니다. 같은 단위, 같은 기준일 위치, 같은 출처 표기 방식이 반복되어야 신뢰가 생깁니다.

## MVP 구현 순서

처음부터 모든 데이터를 실시간으로 연결하려고 하면 범위가 커집니다. 추천 순서는 다음입니다.

1. 정적 샘플 데이터로 첫 화면 완성
2. 지표 데이터 모델과 API 계약 정의
3. 핵심 거시지표 5~8개부터 실제 데이터 연결
4. 경제 캘린더와 발표 상태 연결
5. AI 요약에 근거 지표 링크 붙이기
6. 한국 영향 축 추가
7. 지표 상세 페이지 확장
8. 알림과 즐겨찾기 추가

MVP의 합격 기준은 "모든 기능이 있다"가 아닙니다. 첫 화면에서 미국 경제 상태를 10초 안에 이해하고, AI 요약의 근거를 바로 확인할 수 있으면 MVP로 설득력이 있습니다.

## 검증 기준

이 제품에서 테스트는 단순히 화면이 뜨는지 보는 것이 아닙니다.

- 모든 숫자에 기준일, 단위, 출처, 이전치 대비 변화가 있는가
- AI 요약이 실제 화면의 근거 지표와 연결되는가
- 발표 전/후 경제 일정 상태가 다르게 보이는가
- 한국 영향이 수출, 환율, 금리, 소비, 관세/규제로 분리되는가
- 모바일에서 텍스트와 숫자가 겹치지 않는가
- 출처가 없는 뉴스나 리포트가 AI 요약 근거로 쓰이지 않는가

OpenAI Agents 관점에서는 traces와 evals가 중요합니다. 어떤 도구를 호출했는지, 어떤 지표를 근거로 삼았는지, 어느 단계에서 guardrail이 작동했는지를 기록해야 나중에 품질을 개선할 수 있습니다.
