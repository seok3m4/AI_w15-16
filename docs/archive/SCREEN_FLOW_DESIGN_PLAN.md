# 화면 흐름도 & 디자인 시스템 작성 계획 — 텍스트 기반 Memory MVP

> ⚠️ **아카이브됨 (스테일)**: 이 문서는 화면 흐름도와 디자인 시스템을 *어떻게 만들지*에 대한 작업 계획서다. 산출물([SCREEN_FLOW](../02-frontend/SCREEN_FLOW.md), [DESIGN_SYSTEM](../02-frontend/DESIGN_SYSTEM.md))이 이미 완성되어 더 이상 활성 문서가 아니다. 작성 과정의 의사결정 맥락 보존용으로만 남긴다.

## 1. 문서 목적

이 문서는 `docs/00-product/REQUIREMENTS.md` §12의 후속 산출물 #4 "화면 흐름도"와
디자인 언어(`docs/02-frontend/DESIGN_SYSTEM.md`)를 어떤 순서와 기준으로 작성할지 고정하는 **작업 계획서**다.

ideation/요구사항/API/ERD/아키텍처 5종은 이미 확정되어 있고, 화면 레이어만 비어 있다
(`frontend/`에 코드 없음, `docs/`에 디자인 에셋 없음 — greenfield 상태).

이 작업의 목적은 두 가지다.

1. **화면 흐름도**: 화면 인벤토리 · 네비게이션 · 상태 전이 · 화면↔API/요구사항 매핑을
   고정해 구현자가 같은 그림을 보게 한다.
2. **디자인 언어**: "DESIGN.md만 던지면 예쁜 화면이 안 나온다"는 문제를 해결하기 위해,
   추상 형용사가 아닌 **구체 토큰 + 레퍼런스 앵커 + freeze 전 시각 검증**으로
   디자인 시스템을 못박는다.

## 2. 확정된 방향

- **무드 결정 방식**: 무드를 미리 고르지 않는다. **사용자가 레퍼런스(사이트 링크 또는
  DESIGN.md 예시 파일)를 제공** → 거기서 색/타이포/컴포넌트 언어를 추출 → 실제 HTML로
  스타일 타일 + 대표 화면 1장 렌더 → 피드백 → 반영 후 `docs/02-frontend/DESIGN_SYSTEM.md` freeze.
- **와이어프레임 수준**: 핵심 화면만 저충실도 블록 스케치 (전 화면 고충실도 와이어프레임 아님).
- **범위/구성**: 두 문서, P0~P3 전체(P3는 흐름 위주 경량).

산출물:

- `docs/02-frontend/DESIGN_SYSTEM.md` (디자인 계약서)
- `docs/02-frontend/SCREEN_FLOW.md`

기존 문서 컨벤션 준수: 한국어, 번호 섹션, 표, mermaid, 코드 식별자/API path 원문 유지,
`*_MVP_TEXT_MEMORY.md` 네이밍 패턴.

## 3. 실행 순서

### 3.0 0단계 — 레퍼런스 입력 (사용자 제공, 시작 전제)

사용자가 다음 중 하나 이상을 제공한다.

- 레퍼런스 **사이트 링크**(1~3개) — 좋아하는 제품/페이지
- 또는 **DESIGN.md 예시 파일** / 기존 디자인 토큰 스니펫

함께 받으면 좋은 정보: 라이트/다크 기본값, 포기 못 하는 요소, 피하고 싶은 느낌.
링크가 오면 페이지의 색/타이포/레이아웃 단서를 읽어 분석한다.

### 3.1 1단계 — 레퍼런스 → HTML 렌더 → 피드백 (freeze 전, 가장 먼저)

DESIGN.md를 글로 굳히기 **전에** 취향을 먼저 맞춘다.

- 0단계 레퍼런스에서 색/타이포/컴포넌트 언어를 추출한다.
- 실제 HTML로 다음을 렌더한다.
  - **스타일 타일 1장**: 컬러 팔레트(hex 칩) · 타이포 스케일 샘플(본문/UI, 한글/영문) ·
    버튼/인풋/태그 칩/카드 프리미티브 · shadow/radius 샘플.
  - **대표 화면 1장**: "홈 피드(내 기록)" 또는 "게시물 상세 + AI 근거 인용" 중 1장을
    추출한 언어로 렌더.
- 사용자 피드백(색감/폰트/밀도/온도)을 1~2회 반영한다.
- **이 단계에서 합의된 값이 곧 DESIGN.md의 토큰이 된다.** (역순 금지: 문서 먼저 쓰지 않음)

### 3.2 2단계 — `docs/02-frontend/DESIGN_SYSTEM.md` 작성 (0~1단계 합의값으로 freeze)

> ⚠️ 아래는 **채울 섹션 골격일 뿐**, 색·폰트·무드 등 실제 값은 미리 정하지 않는다.
> 모든 구체값은 0~1단계(레퍼런스 분석 + 렌더 + 피드백)에서 합의된 것만 옮겨 적는다.

1. 문서 목적 · 기준 산출물 링크
2. 디자인 원칙 (3~5개) — 0~1단계에서 합의된 정서/톤으로 작성
3. 레퍼런스 앵커 — 사용자가 준 레퍼런스에서 **가져온 것 / 일부러 안 가져온 것**
4. **디자인 토큰** (전부 구체값, 형용사 금지) — 0~1단계 합의값으로 채움
   - 컬러: 팔레트 + 시맨틱(success/warn/error/info) + (필요 시) 다크 토큰 — 정확한 hex + CSS 변수명
   - 타이포: 폰트(한/영) · 타입 스케일 · line-height · 한글 타이포 기준
   - 스페이싱 · radius · shadow · border · 모션(easing/duration)
5. 컴포넌트 프리미티브 — 버튼/인풋/카드(메모리 포스트 카드)/태그 칩/모달·시트/네비/토스트/
   빈·로딩·에러 패턴/**AI 답변 블록 + 근거 인용(citation) 표시**/승인 대기 카드
6. 레이아웃 그리드 · 반응형 브레이크포인트(웹 기준)
7. 접근성 기준(대비비, 포커스 링, 키보드)
8. 구현 매핑 — CSS 변수 / Tailwind 토큰 형태 제안, 폰트 로딩 방식
9. Do / Don't

### 3.3 3단계 — `docs/02-frontend/SCREEN_FLOW.md` 작성

1. 문서 목적 · 기준 산출물
2. **화면 인벤토리** (표: 화면ID · 이름 · 우선순위 · 주요 API · 접근권한)
3. **전역 네비게이션 맵** (mermaid) — 인증 전/후 영역 분리
4. **핵심 사용자 여정** (mermaid) — 가입→작성→memory 상태→검색→AI요약 / 친구·좋아요 /
   Capsule 생성 / Agent+승인 / MCP
5. **화면별 상세** — 각 화면: 목적 · 진입 경로 · 주요 컴포넌트 · 액션→결과(API 매핑) ·
   상태(빈/로딩/에러/권한없음). **핵심 화면만 저충실도 블록 스케치** 첨부.
   - P0~P2 상세: 로그인/가입, 홈 피드(내 기록), 게시물 상세(좋아요·댓글·AI 근거), 작성/수정,
     키워드 검색 결과, Memory Search(자연어)+AI 요약, 친구 목록/요청, 친구 게시물,
     Capsule 목록/상세/생성, 설정(친구 AI 공유 동의 토글)
   - P3 경량(흐름 위주): Agent 실행/결과/스텝, 승인 대기, MCP 연결 관리
6. 상태/에러 공통 패턴
7. **추적 매트릭스** — 화면 ↔ 요구사항 ID(POST/RAG/CAPSULE/AGENT/MCP…) ↔ API path
8. 남은 검토 항목

저충실도 블록 스케치는 코드펜스 ASCII 블록으로 표현(렌더 의존성 없이 문서에 남도록).

## 4. 재사용할 기존 자산 (새로 만들지 말 것)

- API 엔드포인트: `docs/00-product/REQUIREMENTS.md` §9 + `docs/01-design/API_SPEC.md`
  → 화면별 액션을 여기 path에 그대로 매핑(추측 path 신설 금지)
- 요구사항 ID(AUTH/POST/RAG/CAPSULE/AGENT/MCP…): §7 → 추적 매트릭스 좌측 키로 사용
- 권한 원칙: §6.2 → 화면별 "권한없음" 상태 정의 근거
- 포지셔닝/정서("기억을 다시 꺼내 쓰는"): `docs/00-product/PRODUCT_IDEATION.md`, `README.md` → 디자인 원칙 톤

## 5. 검증 기준

- 스타일 타일/샘플 화면이 사용자가 제공한 레퍼런스와 톤이 맞는지 사용자 확인 통과
- 컬러 토큰 본문/배경 대비가 WCAG AA(본문 4.5:1) 충족
- SCREEN_FLOW의 **모든 화면 액션이 API_SPEC의 실제 path에 매핑**되는지 (미존재 path 0건)
- 요구사항 §7의 P0~P3 기능이 **최소 1개 화면에 대응**되는지 추적 매트릭스로 역검증
- 권한 원칙(§6.2)이 친구/비친구/AI 공유 동의 off 상태 화면에 반영됐는지
- 두 문서가 기존 문서 스타일(한국어·번호 섹션·mermaid·표) 일관성 유지

## 6. 비범위 (Out of scope)

- 풀 픽셀 목업 / Figma 파일 / 전 화면 고충실도 와이어프레임
- 실제 프런트엔드 코드 구현 (`frontend/`)
- 이미지·OCR·멀티모달 화면 (요구사항 §5 제외 범위)
- 모바일 앱 전용 레이아웃 (웹 반응형까지만)
