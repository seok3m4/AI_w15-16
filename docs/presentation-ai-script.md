# 정글 여행 발표 대본

발표 시간 기준: 약 6~8분

## 01. 표지

안녕하세요. 제가 발표할 프로젝트는 **정글 여행**입니다.

정글 여행은 국내 숨은 여행 코스를 지도와 AI로 탐색할 수 있는 게시판 서비스입니다.  
사용자는 직접 다녀온 여행 코스를 장소 순서대로 공유할 수 있고, 다른 사용자는 그 코스를 지도에서 동선으로 확인할 수 있습니다.

여기에 RAG, MCP, Agent 기능을 연결해서 기존 게시글과 실제 장소 정보를 AI 답변에 활용할 수 있도록 만들었습니다.  
기술 스택은 React, NestJS, PostgreSQL과 pgvector, OpenAI, Kakao Maps를 사용했습니다.

## 02. 프로젝트 개요 - 기존 여행 후기의 한계

이 프로젝트는 기존 여행 후기에서 느낀 불편함에서 출발했습니다.

보통 여행 후기를 보면 어떤 장소에 갔는지, 어떤 느낌이었는지는 잘 나와 있습니다.  
하지만 실제로 어디를 먼저 가고, 다음에는 어디로 이동했는지 같은 동선은 한눈에 파악하기 어렵습니다.

그래서 정글 여행에서는 단순히 글을 남기는 게시판이 아니라, 여행 장소의 순서와 지도 흐름까지 함께 볼 수 있는 코스 공유 서비스를 만들고자 했습니다.

핵심 목표는 세 가지였습니다.  
첫째, 여행 코스를 지도 기반으로 공유하는 것.  
둘째, 게시판에 쌓인 글을 AI 검색과 추천에 활용하는 것.  
셋째, AI 답변에 실제 장소 정보까지 연결하는 것입니다.

## 03. 주요 구현 기능 - 게시판 기능과 AI 기능의 결합

먼저 서비스의 기본이 되는 게시판 기능을 구현했습니다.

회원가입과 로그인, 여행 코스 게시글 작성과 수정, 삭제 기능을 만들었고, 코스에 포함되는 경유지를 저장한 뒤 지도에 표시하도록 했습니다.

또한 검색, 태그, 페이징 기능을 통해 게시글을 탐색할 수 있게 했고, 댓글, 좋아요, 저장, 마이페이지 기능도 함께 구현했습니다.

그 위에 AI 기능을 연결했습니다.  
RAG 기반 유사 코스 추천, 게시판 데이터를 활용한 Q&A, MCP를 통한 실제 장소 검색, 그리고 Agent 기반 여행 질문 도우미를 구현했습니다.

중요한 점은 AI 기능을 따로 떨어진 기능으로 만든 것이 아니라, 게시판에 쌓인 여행 데이터를 더 잘 활용하기 위한 흐름으로 연결했다는 점입니다.

## 04. 전체 아키텍처 - NestJS 중심의 전체 구조

전체 구조는 React 프론트엔드, NestJS 백엔드, PostgreSQL 데이터베이스를 중심으로 구성했습니다.

React는 사용자가 보는 게시판 화면, 지도 화면, 마이페이지, AI 질문 화면을 담당합니다.

NestJS는 인증, 게시글, 댓글, 저장 같은 기본 API뿐 아니라 RAG, MCP, Agent 요청까지 처리하는 중심 역할을 합니다.

PostgreSQL에는 사용자와 게시글 데이터가 저장되고, pgvector 확장을 사용해서 게시글 임베딩 데이터도 함께 저장했습니다.

AI 기능은 세 가지 흐름으로 나뉩니다.  
RAG는 게시글 임베딩과 pgvector 검색을 사용해 서비스 내부 데이터를 찾습니다.  
MCP는 Kakao Local API 장소 검색을 별도 도구로 분리합니다.  
Agent는 사용자 질문에 따라 RAG와 MCP를 함께 호출해 최종 답변을 구성합니다.

## 05. RAG 기능 - 게시판 데이터를 지식으로 전환

RAG는 기존 게시글을 AI 답변의 근거로 활용하기 위해 사용했습니다.

사용자가 여행 코스를 작성하거나 수정하면, 제목, 본문, 지역, 태그, 경유지 정보를 하나의 텍스트로 묶습니다.  
그 텍스트를 OpenAI 임베딩 모델로 변환하고, 생성된 벡터를 PostgreSQL의 pgvector에 저장합니다.

이후 사용자가 질문을 하거나 유사 코스를 요청하면, 질문도 다시 임베딩으로 변환합니다.  
그리고 저장된 게시글 벡터와 비교해서 의미가 가까운 코스를 찾습니다.

이렇게 찾은 게시글을 AI 답변의 컨텍스트로 사용하기 때문에, 단순히 일반적인 여행 추천을 하는 것이 아니라 서비스 안에 실제로 존재하는 코스를 근거로 답변할 수 있습니다.

또한 유사 코스 추천에서는 이미 저장된 게시글 벡터를 활용하기 때문에, 기준 게시글과 의미적으로 가까운 다른 코스를 찾아 보여줄 수 있습니다.

## 06. MCP 기능 - 장소 검색 기능의 도구화

MCP는 외부 장소 검색 기능을 AI가 사용할 수 있는 도구 형태로 분리하기 위해 사용했습니다.

이 프로젝트에서는 `place_search`라는 도구를 만들었습니다.  
이 도구는 장소 키워드를 받아 Kakao Local API를 호출하고, 장소 이름, 주소, 좌표, 카카오맵 링크를 반환합니다.

NestJS 서버는 MCP 클라이언트 역할을 합니다.  
필요할 때 MCP 서버의 `place_search` 도구를 호출하고, MCP 서버는 실제 Kakao Local API에 요청을 보내 장소 정보를 가져옵니다.

이 구조의 장점은 장소 검색 로직과 API 키를 백엔드 쪽에 안전하게 둘 수 있다는 점입니다.  
또한 Agent가 필요할 때 표준화된 도구 호출 방식으로 실제 장소 정보를 가져올 수 있습니다.

## 07. Agent 기능 - RAG와 MCP의 조합

Agent는 RAG와 MCP를 함께 사용하는 상위 흐름입니다.

예를 들어 사용자가 "부산에서 바다와 맛집을 같이 즐길 수 있는 코스를 추천해줘"라고 질문했다고 가정하겠습니다.

Agent는 먼저 게시판 안에서 관련 코스를 찾기 위해 `search_similar_posts` 도구를 사용할 수 있습니다.  
이 도구는 RAG 검색을 통해 질문과 의미가 가까운 게시글을 찾습니다.

그 다음 답변에 실제 장소 정보가 필요하면 `place_search` 도구를 호출합니다.  
이때 MCP를 통해 Kakao Local API에서 장소의 주소와 좌표를 확인합니다.

최종적으로 Agent는 사용자 질문에 대한 답변, 참고한 여행 코스 게시글, 실제 장소 정보, 그리고 카카오맵으로 연결할 수 있는 장소 칩을 함께 반환합니다.

정리하면 RAG는 게시판 데이터를 찾는 역할이고, MCP는 외부 장소 정보를 가져오는 역할입니다.  
Agent는 이 둘을 질문 상황에 맞게 선택하고 조합하는 역할을 합니다.

## 08. 데모 - 데모 흐름

데모는 서비스의 핵심 흐름이 자연스럽게 보이도록 구성했습니다.

먼저 로그인한 뒤, 여행 코스 게시글을 작성합니다.  
작성 과정에서 장소 검색을 통해 경유지를 등록하고, 등록된 장소들이 지도에서 어떤 동선으로 보이는지 확인합니다.

그 다음 댓글, 좋아요, 저장 기능을 확인해서 일반 게시판 기능이 동작하는 것을 보여줍니다.

이후 AI 기능으로 넘어가서, 상세 화면에서 유사 코스 추천을 확인합니다.  
마지막으로 AI 여행 질문 도우미 화면에서 질문을 입력하고, Agent가 게시판 데이터와 실제 장소 정보를 활용해 답변하는 흐름을 보여줍니다.

데모에서 가장 보여주고 싶은 핵심은 단순 게시판이 아니라는 점입니다.  
지도 기반 코스 데이터가 서비스 안에 쌓이고, 그 데이터를 AI가 다시 활용한다는 흐름을 보여주는 것이 목표입니다.

## 09. 회고 - AI 기술별 역할 정리

이번 프로젝트를 통해 RAG, MCP, Agent가 각각 다른 역할을 가진다는 점을 확인할 수 있었습니다.

RAG는 서비스 내부 데이터를 AI 답변에 활용하기 위한 기술이었습니다.  
게시판에 쌓인 여행 코스를 임베딩으로 저장하고, 질문과 의미가 가까운 글을 찾는 흐름을 직접 구현했습니다.

MCP는 외부 기능을 AI가 사용할 수 있는 도구로 분리하는 데 의미가 있었습니다.  
장소 검색처럼 외부 API와 연결되는 기능을 독립적인 도구로 만들 수 있었습니다.

Agent는 사용자의 질문에 따라 필요한 도구를 선택하고 조합하는 역할을 했습니다.  
단순히 한 번 답변을 생성하는 것이 아니라, 게시판 검색과 장소 검색을 함께 사용해서 더 구체적인 답변을 만들 수 있었습니다.

## 10. 한계점과 개선 방향

마지막으로 한계점과 개선 방향입니다.

첫 번째 한계는 외부 API 의존도입니다.  
OpenAI와 Kakao API를 호출하기 때문에 비용, 응답 속도, 레이트리밋의 영향을 받을 수 있습니다.

두 번째는 임베딩 캐싱이 부족하다는 점입니다.  
같은 질문이나 비슷한 요청이 반복되어도 매번 임베딩을 새로 생성하는 구조라서 개선 여지가 있습니다.

세 번째는 답변 스트리밍을 적용하지 못했다는 점입니다.  
AI 답변이 한 번에 내려오기 때문에 사용자가 느끼는 대기 시간이 길어질 수 있습니다.

개선 방향으로는 자주 들어오는 질문에 대한 임베딩 캐싱, AI 답변 스트리밍, 긴 게시글 청킹, Agent 도구 확장, 이미지 저장 구조 개선을 생각하고 있습니다.

이번 프로젝트는 AI 기능을 단순히 붙이는 것보다, 서비스 데이터와 외부 도구를 어떻게 연결할 수 있는지 실험했다는 점에서 의미가 있었습니다.  
이상으로 발표를 마치겠습니다. 감사합니다.
