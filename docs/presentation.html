04 — AI 기능 · RAG
🧠 Retrieval-Augmented Generation

내 DB 데이터를
근거로 답하게 만들기

기능

  • 유사 코스 추천 · 저장된 벡터로 즉시 비교 (OpenAI 호출 0번)
  • 시맨틱 검색 · 자유 텍스트 → 가까운 코스
  • Q&A · 후기를 컨텍스트로 답변 + 출처 표시

기술

  • 본문을 1536차원 임베딩 → pgvector 저장
  • 코사인 거리 <=> 로 의미 검색 (raw SQL)
  • 프레임워크 없이 openai SDK + pgvector 직접 구현
① 저장 (글 작성 시, 미리) 본문 → OpenAI 임베딩 → vector(1536) → PostEmbedding 저장 ② 검색·생성 (질문 시) 질문 → 질문 벡터 → pgvector 코사인 검색 (Retrieval) → 가까운 코스 N개 → 컨텍스트로 OpenAI 채팅 (Generation) → 답변 + 출처 [1][2]
04 — AI 기능 · MCP
🔌 Model Context Protocol

외부 능력을
"표준 도구"로 분리

기능

  • Kakao 장소 검색을 표준 tool로 노출
  • AI Agent가 실제 장소의 좌표(lat/lng)를 가져옴

기술

  • 독립 프로세스 · 공식 @modelcontextprotocol/sdk (stdio)
  • 도구 place_search · 키워드 → 이름·주소·좌표
  • NestJS가 자식 프로세스로 띄워 MCP 클라이언트 연결
  • REST 키는 서버에만 · 브라우저 비노출
왜 별도 프로세스인가 도구를 표준 프로토콜로 분리 → 다른 AI 클라이언트도 재사용 ─────────────── NestJS (MCP 클라이언트) │ stdio ▼ MCP 서버 (place_search) │ HTTPS ▼ Kakao Local API → 이름 · 주소 · 좌표 GET /mcp/places?q=해운대
04 — AI 기능 · Agent
🤖 Function Calling Agent

모델이 도구를
스스로 골라 반복 호출

기능

  • 여행 질문 → 기존 후기 + 실제 장소를 함께 근거로 답변
  • MCP와 RAG를 도구로 함께 쓰는 상위 기능

기술

  • 도구 2종 · search_similar_posts(RAG) + place_search(MCP)
  • 상태는 OpenAI messages 배열에 누적
  • MAX_ROUNDS=5로 무한 루프 방지 + fallback
  • 도구 결과 슬림화로 토큰 폭증 방지
"부산 바다+맛집 코스 추천해줘" │ POST /agent/ask ▼ ┌─ AgentService 루프 ──────┐ │ ① search_similar_posts │ RAG │ → 기존 부산 코스 │ │ ② place_search │ MCP │ → 실제 장소 좌표 │ │ ③ 최종 답변 생성 │ └──────────────────────────┘ │ ▼ 답변 + 참고 게시글 + 카카오맵 장소 칩 RAG=정해진 1단계, Agent=모델이 판단

06 — 회고 · 한계 · 개선

세 AI 기술의 역할 분담이 명확해졌다

✅ 회고

  • 역할 분담: MCP=도구, RAG=내 데이터 검색·생성, Agent=조합·판단
  • 프레임워크 없이 RAG 직접 구현 → 동작 원리 투명
  • pgvector를 기존 PG에 얹어 인프라 단순 유지

⚠️ 한계점

  • 외부 API 의존: 한↔미 RTT·비용·레이트리밋, Agent는 수 초 소요
  • 질문 임베딩 매번 생성 (캐싱 미적용)
  • 스트리밍 미적용 + 본문 8000자 제한
  • 이미지 base64 DB 저장으로 응답 비대

🚀 개선 아이디어

  • 같은 리전 배포 + 답변 스트리밍
  • 질문 임베딩 캐싱(Redis)
  • 긴 본문 청킹 후 다중 벡터
  • Agent 도구 확장 · 동선/예산 최적화
  • 이미지 S3 분리로 경량화
