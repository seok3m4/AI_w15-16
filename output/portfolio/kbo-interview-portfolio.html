05

핵심 기능

경기방은 경기 전·중·후가 끊기지 않아야 합니다

경기방은 이번 프로젝트의 중심 화면입니다. 데이터가 없을 때도 빈 화면 대신 현재 상태를 알려주는 데 신경 썼습니다.

경기 전
경기 시간, 양 팀, 선발투수, 라인업을 보여줍니다.
경기 중
스코어와 1회부터 9회까지 문자중계를 확인할 수 있습니다.
경기 후
박스스코어, 타자 기록, 투수 기록, 승리/패전/세이브 투수를 보여줍니다.
참여
관련 글 요약, 리뷰 작성, 댓글과 추천으로 커뮤니티 흐름을 이어갑니다.

06

트러블슈팅 1

유사 글 검색은 DB 환경부터 맞춰야 했습니다

RAG 기능을 붙이는 과정에서 pgvector 확장과 배포 DB 설정이 맞지 않으면 기능이 시작되지 않는다는 것을 경험했습니다.

문제
로컬 PostgreSQL에 pgvector 확장이 없어 migration이 실패했습니다.
시도
문자열 저장, 별도 Vector DB, PostgreSQL 통합 방식을 비교했습니다.
해결
로컬과 Supabase DB에 vector 확장을 활성화하고 게시글 임베딩 저장 흐름을 만들었습니다.
배운 점
AI 기능은 모델 호출뿐 아니라 DB 확장, migration, 배포 환경까지 같이 맞아야 합니다.
KBO Talk / KBO Fan Hub
07

트러블슈팅 2

외부 데이터는 없을 때까지 고려해야 했습니다

KBO 경기 데이터는 경기 상태에 따라 선발, 라인업, 기록 공개 여부가 달랐습니다.

문제
예정 경기, 진행 중 경기, 종료 경기마다 가져올 수 있는 데이터가 달랐습니다.
시도
라인업 공개 전/후, 종료 경기, 오늘 경기, 과거 경기를 나눠 확인했습니다.
해결
라인업 미공개, 기록 준비 중, 종료 경기 상태에 따라 버튼과 안내 문구를 다르게 처리했습니다.
배운 점
외부 연동은 데이터를 가져오는 코드보다 데이터가 없을 때의 UX가 더 중요했습니다.
KBO Talk / KBO Fan Hub
08

기술 스택

기술은 선택 이유와 사용 경험으로 설명했습니다

단순 나열 대신, 어떤 문제를 해결하기 위해 어떤 기술을 썼는지 적었습니다.

Next.js / React
웹 화면과 API를 한 프로젝트에서 관리하고, 게시판·경기방·모바일 앱형 화면을 컴포넌트로 나눴습니다.
TypeScript / Prisma
게시글, 댓글, 경기 데이터처럼 구조가 있는 값을 타입과 DB schema로 연결했습니다.
PostgreSQL / pgvector
게시판 데이터와 유사 글 검색용 임베딩을 하나의 DB에서 다루기 위해 사용했습니다.
OpenAI / LangChain.js
비슷한 글, 관련 글 요약, 리뷰 초안처럼 글쓰기 주변 기능을 구현했습니다.
Vercel / Supabase / Android WebView
웹 배포, 운영 DB, 모바일 앱형 APK까지 실제로 확인 가능한 산출물로 연결했습니다.
